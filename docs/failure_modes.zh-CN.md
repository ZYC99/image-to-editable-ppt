# Failure Modes 中文版

本文档记录 Image-to-Editable-PPT 项目中出现过或预期会出现的失败模式。

## FM-001：PowerPoint 打开提示修复

### 现象

生成的 `.pptx` 用 PowerPoint 打开时提示“PowerPoint 发现文件中的内容有问题，需要修复”。

### 影响

这是阻断级失败。  
只要出现这个问题，就不能继续讨论视觉相似度或可编辑性。

### 可能原因

- PPTX 内部 zip 结构损坏。
- 关系文件 `.rels` 不完整。
- slide XML 中存在 PowerPoint 不兼容内容。
- 手写 OpenXML 出错。
- 使用了不稳定的 shape、shadow、gradient 或 transform 配置。
- 图片引用路径错误。

### 处理策略

- 使用 `python-pptx` 等稳定库生成基础对象。
- 避免手写复杂 OpenXML。
- 先通过 zip integrity 检查。
- 用 LibreOffice headless 尝试打开/导出作为近似检查。
- 复杂效果先降级为普通形状或裁剪图片。

### 规则沉淀

对应 Rule 001：兼容性优先。

## FM-002：整页截图假装可编辑

### 现象

PPT 看起来很像原图，但实际上只是把整张图片放进页面，最多叠加少量文本框。

### 影响

这不符合 image-to-editable-PPT 的目标。  
除非用户明确选择 `fidelity_first`，否则不能算合格输出。

### 可能原因

- 为了快速提高相似度，把整页图片作为背景。
- renderer 无法稳定重建复杂元素。
- 没有 editable gate 检查。

### 处理策略

- 默认禁止整页截图作为主要内容。
- 检查大图面积占比。
- 检查文本框数量和形状数量。
- 标题、关键数字、说明文字必须可编辑。
- 复杂元素可以局部裁剪，但不能整页裁剪。

### 规则沉淀

对应 Rule 002：禁止整页截图假装可编辑。

## FM-003：中文文本框溢出或换行错误

### 现象

源图中文字一行显示，但生成 PPT 后变成两行、被裁切、行距过大或位置偏移。

### 影响

会严重影响中文商业 PPT 的可用性。

### 可能原因

- PowerPoint 的中文字体渲染与脚本估算不同。
- 字体缺失，被系统替换。
- 文本框宽度不足。
- 行距、段前段后、自动换行策略不一致。
- 不同系统 PowerPoint 版本渲染不同。

### 处理策略

- 给文本框预留更大宽度。
- 设置合理的 `autofit` 或 shrink 策略。
- 建立中文字体 fallback 列表。
- 优先使用常见中文字体。
- 输出后通过截图或人工反馈修正。

### 规则沉淀

需要在 Layout IR 中加入 `fit`、`font_fallbacks`、`min_font_size` 等字段。

## FM-004：图标用 PPT 原生形状硬画导致不像

### 现象

源图里的图标很精致，但生成 PPT 后用圆、线、矩形拼出来，视觉差距很大。

### 影响

整体高级感明显下降。

### 可能原因

- 复杂图标不适合用基础形状重建。
- 缺少 SVG/icon library。
- 模型误判图标为可重建形状。

### 处理策略

- 简单图标可以 rebuild。
- 复杂图标优先 `asset_search` 或 `crop`。
- 保留图标区域为局部 PNG。
- 后续接入本地 SVG 素材库。

### 规则沉淀

复杂低编辑价值视觉元素不要硬画，默认走 crop/SVG。

## FM-005：阴影、渐变和透明度还原差

### 现象

卡片、背景、按钮的阴影和渐变效果与原图差异明显，页面显得扁平。

### 影响

视觉质感下降，但通常不影响内容编辑。

### 可能原因

- PPT 原生阴影和图片生成中的阴影不是同一种渲染模型。
- 透明度叠加、模糊、混合模式难以精确还原。
- 渲染器能力有限。

### 处理策略

- 基础卡片阴影使用简化参数。
- 复杂光效作为背景裁剪。
- 对低编辑价值装饰元素使用 `background_crop`。
- 不为了还原阴影而破坏 PPTX 兼容性。

## FM-006：布局整体像，但细节间距不准

### 现象

页面结构基本正确，但卡片间距、文字位置、图标大小、圆角等细节与原图不一致。

### 影响

第一眼能看出不是同一张设计稿。

### 可能原因

- 视觉解析 bbox 不够准。
- 坐标从 px 到 inch 映射有误差。
- 字体渲染导致视觉占位变化。
- 缺少 screenshot diff/refine loop。

### 处理策略

- 保持统一坐标体系。
- 增加 layout IR 中的 alignment/group 信息。
- 使用截图对比进行二次修正。
- 对常见布局模式建立模板化规则。

## FM-007：可编辑性高但保真度低

### 现象

所有元素几乎都能编辑，但页面不像原图，尤其图标、装饰、复杂背景效果差。

### 影响

适合可编辑优先场景，但不适合正式汇报交付。

### 可能原因

- 过度使用 PPT 原生对象。
- 没有裁剪复杂区域。
- 没有区分高编辑价值和低编辑价值元素。

### 处理策略

- 默认使用 `balanced`。
- 复杂视觉元素允许局部裁剪。
- 用策略层为每个元素分配 editability_score 和 reconstructability_score。

## FM-008：保真度高但可编辑性低

### 现象

看起来很像原图，但大量内容都是图片，无法编辑。

### 影响

不符合多数用户对 editable PPT 的期待。

### 可能原因

- 过度使用 crop。
- 没有 editable gate。
- 没有区分文本和背景。

### 处理策略

- 使用 `text_on_crop`：背景裁剪，文字重建。
- 检查文本框数量。
- 检查关键文本是否可编辑。
- 仅在 `fidelity_first` 模式允许大量图片化。
