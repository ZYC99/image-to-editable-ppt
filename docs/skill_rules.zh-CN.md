# Skill Rules 中文版

本文档记录 Image-to-Editable-PPT 项目的核心规则。  
这些规则用于约束 ChatGPT、Codex 和本地 toolchain 的行为。

## Rule 001：兼容性优先

生成的 `.pptx` 必须能在 PowerPoint 中正常打开，不能提示“需要修复”。

如果 PowerPoint 打开文件时提示修复，则本轮输出直接判定失败，不进入视觉相似度评估。

### 要求

- 输出前检查 PPTX 是否为合法 zip 文件。
- 检查核心 PowerPoint XML 文件是否存在。
- 尽量使用稳定库生成 PPTX，例如 `python-pptx`。
- 避免手写危险 OpenXML。
- 复杂效果优先降级，而不是生成不稳定结构。

## Rule 002：禁止整页截图假装可编辑

除非用户明确选择 `fidelity_first` 模式，否则不能把整页原图作为唯一主要内容。

整页截图背景 + 少量文本框，不算合格的 image-to-editable-PPT。

### 要求

默认 `balanced` 模式下：

- 主标题必须可编辑。
- 副标题必须可编辑。
- 关键数字必须可编辑。
- 核心说明文字必须可编辑。
- 主要卡片和形状尽量可编辑。
- 整页大图占比必须受到限制。

## Rule 003：关键内容必须可编辑

用户最可能修改的内容必须优先重建为 PPT 原生对象。

### 优先可编辑对象

- 主标题
- 副标题
- 页码/章节标签
- KPI 数字
- 指标名称
- 结论文字
- 表格中的主要文字
- 流程节点文字
- 卡片标题
- 说明性正文

### 可以裁剪的对象

- 复杂图标
- 复杂插画
- 背景纹理
- 光效
- 阴影叠加
- 不需要编辑的装饰元素
- 无法可靠识别数据的复杂图表

## Rule 004：默认使用混合策略

默认模式是 `balanced`，不是纯保真，也不是纯可编辑。

推荐策略：

| 元素 | 默认策略 |
|---|---|
| 文本 | rebuild |
| 卡片/矩形/圆角矩形 | rebuild |
| 线条/分割线/箭头 | rebuild |
| 简单图标 | SVG 或 shape rebuild |
| 复杂图标 | crop 或 asset_search |
| 背景装饰 | crop 或简化重建 |
| 插画 | crop |
| 简单图表 | native_chart 或 rebuild |
| 复杂图表 | crop + 可编辑说明文字 |

## Rule 005：必须输出验证报告

每次生成 PPTX 后，都应该输出 `validation_report.json`。

报告至少包含：

- PPTX 是否为合法 zip
- 核心 XML 是否存在
- 文本框数量
- 形状数量
- 图片数量
- 是否存在疑似整页截图
- 可编辑对象占比
- 关键元素是否可编辑
- 是否通过 MVP editable gate
- 失败原因列表

## Rule 006：失败必须沉淀

每次出现失败，都要记录到 `docs/failure_modes.zh-CN.md`。

失败记录应该包括：

- 失败编号
- 现象
- 可能原因
- 处理策略
- 是否已经修复
- 对后续规则的影响

## Rule 007：模型只负责理解，不负责最终精确渲染

ChatGPT 或视觉模型可以负责：

- 分析图片
- 拆分页面结构
- 生成 layout IR
- 判断元素策略
- 解释失败原因
- 提出修正建议

但最终 `.pptx` 应由稳定 renderer 生成，不应该让模型直接自由生成复杂 OpenXML。

## Rule 008：先稳定，再变漂亮

项目开发顺序应该是：

1. 文件能正常打开。
2. 可编辑对象数量达标。
3. 主要内容位置正确。
4. 视觉相似度提升。
5. 加入截图 diff。
6. 加入自动 refine。

不要一开始就追求 95% 视觉复刻。
